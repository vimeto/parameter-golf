# Autoresearch Experiment Skill

## What This Does
Continuous two-tier experiment loop: quick exploration sweeps on packed 1-GPU nodes, validated by full 8-GPU runs at token-parity with the competition benchmark. Both nodes must be running at all times.

## Code Architecture

### Root `train_gpt.py` = Template
- Contains ONLY proven, graduated features (ROCm, torch.compile, base model, eval pipeline)
- All experiment copies start from this base
- Features merge into root only after validation confirms improvement

### Per-Experiment Scripts
- Each exploration run gets its own `train_gpt.py` copy at `specs/batchN/runM.py`
- Generated by subagents from root, with specific modifications per experiment
- Referenced in spec files via `TRAIN_SCRIPT=specs/batchN/runM.py`
- Validation runs also use `TRAIN_SCRIPT` pointing to the winning experiment's script

## Two-Tier Architecture

### Node A — Exploration (quick screening)
- Platform: 8x 1-GPU MI250X packed into one node, 50 min training + 10 min eval
- Token budget: ~360M tokens per run (directional signal only)
- Submit: `scripts/lumi.sh submit scripts/slurm/lumi_packed.sh specs/batchN/spec.txt`
- Each run has its own train_gpt.py with a different modification

### Node B — Validation (real numbers)
- Platform: 1x 8-GPU MI250X full node, 45 min training (MAX_WALLCLOCK_SECONDS=2700)
- Token budget: ~6.8B tokens (token-parity with 8xH100 benchmark)
- Submit: `TRAIN_SCRIPT=specs/batchN/runM.py NUM_GPUS=8 scripts/lumi.sh submit scripts/slurm/lumi_train.sh`
- Uses the winning exploration experiment's script

Both tiers run in parallel continuously. While validation runs, the next exploration batch is already submitted.

## The Main Loop

```
1. Read experiments/state.json for current state
2. Check running jobs: scripts/lumi.sh status
3. Process completed exploration batch:
   a. scripts/lumi.sh packed-results <job_id> for compact results
   b. Verify integrity of each run
   c. Rank runs, write to experiments/exploration_log.md
   d. Pick best run for promotion to validation
   e. Submit validation run on Node B (TRAIN_SCRIPT=winning script)
4. Process completed validation run:
   a. Parse results, compare to global best
   b. Write to experiments/validation_log.md
   c. If new best: graduate winning changes into root train_gpt.py
   d. Update experiments/state.json
5. Generate next exploration batch (4-step process below)
6. Submit exploration on Node A
7. Update experiments/state.json with new job IDs
```

## Exploration Batch Generation (4 Steps)

### Step 1: Data-Driven Analysis
- Read state.json, exploration_log.md, validation_log.md
- What worked? What failed? What's untested?
- Check TODO.md for phase priorities
- Identify the highest-impact research direction

### Step 2: Conceptual Run List
Design 8 experiments with clear hypotheses:
- Run 0: **Control** — reproduce current best for baseline comparison
- Runs 1-6: **Variations** — each tests one hypothesis from the analysis
- Run 7: **Wildcard** — something creative or high-risk/high-reward

Document the rationale for each run BEFORE generating any code.

### Step 3: Generate Scripts via Subagents
- Launch subagents in parallel (one per script or batched)
- Each subagent: reads root `train_gpt.py` → applies the specific modification → writes to `specs/batchN/runM.py`
- Each script must be self-contained (no env var dependencies for the experimental feature)
- The experimental change should include a log line for integrity verification

### Step 4: Validate and Submit
- Syntax check each script: `python3 -c "import ast; ast.parse(open('specs/batchN/runM.py').read())"`
- Create spec file `specs/batchN/spec.txt` with all 8 runs
- `git add specs/batchN/ && git commit -m "feat: exploration batch N"`
- `scripts/lumi.sh sync`
- `scripts/lumi.sh submit scripts/slurm/lumi_packed.sh specs/batchN/spec.txt`

## Rules

- NEVER modify root `train_gpt.py` for experimentation — only for graduating proven features
- NEVER modify `records/our_submission/` — it is a frozen snapshot
- NEVER modify evaluation logic or BPB calculation in experiment scripts
- Follow research directions in `~/code/omat/parameter-golf/TODO.md`
- Always check **post-quantization** bpb, not pre-quant
- MI250X exploration results are directional — validation at full scale is authoritative
- Use 1-line commit messages, NO co-authorship

## Startup Crash Policy

- **1 run crashes at startup**: let other 7 continue, note failed run for next batch
- **2+ runs crash at startup**: cancel node (`scripts/lumi.sh cancel <job_id>`), fix bug, resubmit all 8
- **Run diverges mid-training**: let it run, data is still informative

## Validation Logging Format

```
## Validation N (job XXXXX) — <run_id> (from batch M)
Script: specs/batchM/runN.py
Config: <key env vars>
Result: **X.XXXX bpb** | artifact XX.XMB | N steps in Ns
vs global best: X.XXXX -> X.XXXX [NEW BEST / worse]
Action: graduate to root / discard / try runner-up
```

## Graduating Features to Root

When a validation run beats the global best:
1. Diff the winning script against root `train_gpt.py`
2. Merge the winning changes into root
3. Commit: `git commit -m "feat: graduate <feature> from batch N validation"`
4. Update `experiments/state.json` with new global best
5. All future experiment copies now inherit this improvement

## Phase Advancement

- Phase complete when: 3+ validation batches show <0.005 bpb improvement, OR all TODO items tried
- Advance to next phase, carry forward best config as new baseline
- Update `experiments/state.json` with new phase number

## Uncertain Category

Some techniques (late QAT, progressive training, curriculum learning) may only show benefit at full token budget:
- Mark as "uncertain" in exploration results
- Test only in validation-length runs
- Flag for human review

## Recording Results

After each validation run, append to `experiments/results.tsv`:
```
exp_id	date	description	val_bpb	artifact_bytes	notes
```
